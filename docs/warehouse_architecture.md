# PostgreSQL Data Warehouse Architecture

## Purpose

The PostgreSQL warehouse is designed as the analytical backbone of the Retail Intelligence Platform. Its role is to convert local CSV outputs into governed, queryable, business-ready tables that can support reporting, forecasting, and application APIs.

The architecture separates ingestion, cleaning, dimensional modeling, and consumption into clear database layers. This keeps the project scalable, auditable, and easier to evolve as new retail datasets or business requirements are added.

## End-to-End Flow

```text
CSV Files
   |
   v
raw schema
   |
   v
staging schema
   |
   v
warehouse schema
   |
   v
marts schema
   |
   +--> Power BI
   +--> Machine Learning
   `--> FastAPI
```

## Layer Design

### 1. CSV Files

CSV files are the source landing format for this project. They include original Kaggle datasets and locally generated cleaned outputs.

Source data remains outside Git because large raw and processed datasets should not be committed to the repository.

Expected local zones:

- `data/raw/`: original downloaded datasets.
- `data/processed/`: cleaned outputs generated by Python scripts.
- `data/external/`: optional reference files or enrichment data.

### 2. `raw` Schema

The `raw` schema stores source-aligned tables loaded from CSV files with minimal transformation.

Business role:

- Preserve the source shape as closely as possible.
- Support traceability back to the original datasets.
- Provide a reloadable landing area for data ingestion.

Design principles:

- Use source-oriented table names.
- Avoid destructive transformations.
- Keep ingestion metadata where useful, such as load timestamp or source file name.
- Treat this layer as appendable or reloadable depending on the source.

Example table direction:

- `raw.online_retail`
- `raw.walmart_sales_train`
- `raw.walmart_sales_test`
- `raw.walmart_stores`
- `raw.walmart_features`

### 3. `staging` Schema

The `staging` schema standardizes and validates raw data before it enters the warehouse model.

Business role:

- Normalize column names.
- Cast data types.
- Remove invalid records.
- Prepare consistent business keys.
- Apply documented data quality rules.

Design principles:

- Use clean, typed, source-specific tables.
- Keep transformations transparent and easy to test.
- Do not aggregate data in staging.
- Create one staging table per major source entity where possible.

Example table direction:

- `staging.stg_online_retail_transactions`
- `staging.stg_walmart_sales`
- `staging.stg_walmart_stores`
- `staging.stg_walmart_features`

### 4. `warehouse` Schema

The `warehouse` schema contains the conformed dimensional model. This is the enterprise-style analytics layer where facts and dimensions are designed around business processes.

Business role:

- Provide stable tables for analytics and BI.
- Centralize business definitions.
- Support reusable facts and dimensions.
- Enable consistent reporting across tools.

Design principles:

- Use surrogate primary keys for dimensions.
- Store natural source keys as attributes for traceability.
- Keep fact tables at a clearly defined grain.
- Use foreign keys from facts to dimensions.
- Avoid dashboard-specific calculations in core warehouse tables.

Core model:

- `warehouse.fact_sales`
- `warehouse.dim_date`
- `warehouse.dim_product`
- `warehouse.dim_customer`
- `warehouse.dim_store`

### 5. `marts` Schema

The `marts` schema contains curated, consumption-ready tables and views for specific business use cases.

Business role:

- Simplify access for BI dashboards.
- Provide KPI-ready views for analysts.
- Serve ML feature tables when appropriate.
- Separate consumption logic from the core warehouse model.

Design principles:

- Build marts from `warehouse` tables, not directly from raw files.
- Keep naming business-readable.
- Create separate marts for reporting, forecasting, and API consumption when needed.
- Optimize for the consuming tool without changing warehouse truth tables.

Example table direction:

- `marts.sales_overview`
- `marts.customer_performance`
- `marts.product_performance`
- `marts.store_performance`
- `marts.forecasting_features`

## Consumption Layer

### Power BI

Power BI should connect to the `marts` schema for dashboard-ready views. This keeps report logic simple and avoids embedding complex transformations inside the dashboard.

Expected use cases:

- Executive sales overview.
- Store performance.
- Product performance.
- Customer behavior.
- Time-series sales trends.

### Machine Learning

Machine learning pipelines should use curated feature tables from `marts` or carefully selected warehouse tables.

Expected use cases:

- Sales forecasting.
- Demand pattern analysis.
- Feature importance review.
- Model performance tracking.

### FastAPI

FastAPI can expose selected metrics and predictions from the `marts` schema or from model output tables.

Expected use cases:

- KPI endpoints.
- Store or product performance endpoints.
- Forecast endpoints.
- Dashboard backend services.

## Production-Oriented Practices

- Keep raw data immutable where possible.
- Document every cleaning rule before loading into warehouse tables.
- Use repeatable SQL scripts and version control for schema changes.
- Separate source-specific staging from conformed warehouse modeling.
- Add data quality checks before facts and dimensions are refreshed.
- Prefer marts/views for dashboard logic instead of changing core warehouse tables.
- Track refresh timestamps and source lineage as the project matures.
